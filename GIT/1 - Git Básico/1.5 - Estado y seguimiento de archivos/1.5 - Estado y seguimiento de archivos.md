
#git #estado #seguimiento #git-status

---

## 1. ¿Qué es el estado y seguimiento de archivos?

El **estado y seguimiento de archivos** en Git describe cómo Git controla los cambios de los archivos dentro de un repositorio y en qué punto del flujo de trabajo se encuentran.

- Define si un archivo está siendo controlado por Git o no  
- Permite saber si un archivo fue modificado, preparado o confirmado  
- Es la base para entender cómo funcionan los commits  

Sin este concepto, no se puede usar Git de forma consciente ni segura.

---

## 2. ¿Para qué sirve el estado de los archivos?

El estado de los archivos sirve para:

- Saber qué archivos están listos para un commit  
- Detectar cambios no guardados o no preparados  
- Evitar commits incompletos o accidentales  
- Controlar qué información entra al historial del proyecto  

Es una herramienta de **control y visibilidad** del trabajo.

---

## 3. Estados posibles de un archivo en Git

Git maneja un **ciclo de vida** para cada archivo dentro del repositorio:

1. **Untracked (No trackeado)**  
   - Git no lo está controlando  
   - Aparece cuando se crea un archivo nuevo  

2. **Tracked (Trackeado)**  
   Un archivo trackeado puede estar en varios estados:
   - **Unmodified** → no tiene cambios  
   - **Modified** → fue editado pero no preparado  
   - **Staged** → está listo para el commit  

Este ciclo se repite constantemente durante el desarrollo.

---

## 4. ¿Qué significa que un archivo esté trackeado?

Un archivo **trackeado** es aquel que Git conoce y supervisa.

- Git registra sus cambios  
- Puede formar parte de commits  
- Aparece en el historial del repositorio  

Un archivo pasa a estar trackeado cuando se lo agrega por primera vez al **stage**.

---

## 5. ¿Qué significa que un archivo no esté trackeado?

Un archivo **no trackeado** es aquel que:

- Existe en el directorio del proyecto  
- Git lo detecta, pero no lo controla  
- No forma parte de ningún commit  

Git informa estos archivos para que el desarrollador decida:
- Agregarlos al repositorio  
- O ignorarlos explícitamente  

---

## 6. El rol del área de staging

El **staging area** (área de preparación) actúa como un filtro entre el trabajo y el historial.

- Permite elegir qué cambios se van a commitear  
- Evita subir cambios innecesarios  
- Da control total sobre cada commit  

No todo lo modificado debe ir automáticamente a un commit.

---

## 7. El comando git status

`git status` es el comando principal para entender el estado del repositorio.

Permite ver:
- Archivos modificados  
- Archivos en staging  
- Archivos no trackeados  
- El estado general de la rama  

Es el comando más usado en el flujo diario con Git.

---

## 8. Flujo básico de estado y seguimiento

El flujo típico es:

1. Crear o modificar archivos  
2. Ver el estado con `git status`  
3. Preparar cambios para commit  
4. Confirmar cambios  

Este ciclo se repite constantemente durante el desarrollo.

---

## 9. Buenas prácticas

- Ejecutar `git status` antes de cada commit  
- No agregar archivos innecesarios al stage  
- Mantener commits pequeños y claros  
- Usar `.gitignore` para archivos que no deben versionarse  

---

## 10. Conceptos clave

- Estado del archivo  
- Trackeado / No trackeado  
- Modified / Staged / Committed  
- Staging area  
- git status  

---

## 11. Resumen final

El estado y seguimiento de archivos es uno de los conceptos más importantes de Git.

Permite:
- Entender qué está pasando en el repositorio  
- Tener control total sobre los commits  
- Evitar errores comunes en el versionado  

Dominar este tema es clave para trabajar de forma profesional con Git.
