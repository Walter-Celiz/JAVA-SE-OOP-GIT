---

## 1. Definición

Los **tags en Git** son referencias especiales que apuntan a un commit específico y se utilizan principalmente para **marcar versiones importantes** del proyecto, como lanzamientos (`v1.0`, `v2.1`, etc.).

A diferencia de las ramas, los tags **no avanzan** con nuevos commits: quedan fijos en el punto exacto del historial que representan.

---

## 2. Propósito

El uso de tags permite:
- Identificar versiones estables del proyecto.
- Facilitar la gestión de releases.
- Volver fácilmente a un estado específico del código.
- Mantener un historial claro de versiones publicadas.

Son fundamentales en flujos profesionales y proyectos versionados.

---

## 3. Funcionamiento interno

Internamente, un tag es una referencia (`refs/tags/`) que apunta a un commit SHA.

Existen dos tipos principales:
- **Tags ligeros (lightweight)**: simples punteros a un commit.
- **Tags anotados (annotated)**: objetos completos que almacenan:
  - autor
  - fecha
  - mensaje
  - checksum

Git trata los tags como referencias inmutables salvo que se eliminen explícitamente.

---

## 4. Tipos de tags
### Tag ligero
- No contiene metadata adicional.
- Similar a una rama fija.
- Uso rápido y simple.

### Tag anotado
- Recomendado para versiones oficiales.
- Incluye mensaje y datos del autor.
- Puede firmarse con GPG.

## 6. Aclaraciones importantes

- Los tags **no se suben automáticamente** al repositorio remoto.
- Es necesario hacer `push` explícito de los tags.
- Un tag no cambia aunque la rama siga avanzando.
- Pueden eliminarse y recrearse, pero **no es buena práctica** en versiones publicadas.

---

## 7. Ejemplo conceptual

Un flujo típico sería:
1. Se termina una funcionalidad importante.
2. Se hace commit final.
3. Se crea un tag `v1.0` apuntando a ese commit.
4. Ese tag representa una versión estable del proyecto.

Aunque luego se sigan agregando commits, `v1.0` siempre apuntará al mismo estado del código.

---

## 8. Ventajas

- Historial claro y ordenado.
- Fácil acceso a versiones antiguas.
- Mejora la comunicación en equipos.
- Base para sistemas de release automatizados.

---

## 9. Buenas prácticas

- Usar **tags anotados** para releases oficiales.
- Seguir versionado semántico (`vMAJOR.MINOR.PATCH`).
- No modificar tags publicados.
- Documentar qué representa cada versión.

---

## 10. Notas

- GitHub y GitLab utilizan tags para crear releases.
- Un tag puede apuntar a cualquier commit, no solo a `main`.
- Se pueden listar, borrar y subir selectivamente.

## 12. Resumen final

Los tags en Git permiten marcar puntos importantes del historial, generalmente versiones del proyecto. Son referencias fijas, claras y fundamentales para el versionado profesional y la gestión de releases en proyectos reales.
