--- 
## Definición

`git rebase` es una operación que permite **reaplicar commits de una rama sobre otra**, reescribiendo el historial para que sea más **lineal y limpio**.

A diferencia de `merge`, rebase **no crea un commit de unión**, sino que mueve los commits como si se hubieran creado a partir de otro punto.

---

## Propósito

El objetivo principal del rebase es:

- Mantener un historial de commits **simple y lineal**
- Evitar commits de merge innecesarios
- Facilitar la lectura del historial del proyecto

---

## Cómo funciona internamente

Cuando ejecutas un rebase:

1. Git guarda temporalmente los commits de tu rama
2. Mueve la rama base al nuevo punto
3. Reaplica uno por uno los commits guardados
4. Genera **nuevos commits** (con nuevos hashes)

> Importante: el historial se **reescribe**

---

## Rebase vs Merge

| Rebase                   | Merge                        |
| ------------------------ | ---------------------------- |
| Historial lineal         | Historial con bifurcaciones  |
| Reescribe commits        | Conserva commits originales  |
| No crea commit extra     | Crea commit de merge         |
| Ideal para ramas locales | Ideal para ramas compartidas |

---

## Cuándo usar rebase

✔ Antes de integrar una rama local  
✔ Para limpiar commits antes de subirlos  
✔ Para mantener ramas de trabajo actualizadas  

❌ En ramas compartidas con otros  
❌ Sobre commits ya publicados  

---

## Buenas prácticas

- Usar rebase **solo en ramas locales**
- Nunca rebasear `main` o `develop`
- Si hay conflictos, resolverlos con cuidado
- Entender que los hashes cambiarán

---

## Resumen

- Rebase reescribe el historial
- Produce un flujo más limpio que merge
- Es potente, pero peligroso si se usa mal
- Ideal para trabajo individual
