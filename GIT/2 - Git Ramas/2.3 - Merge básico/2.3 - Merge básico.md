## 1. Definición

El **merge básico** es el proceso mediante el cual Git **une los cambios de una rama con otra**, integrando el historial de trabajo realizado en paralelo dentro de un mismo proyecto.

Es una operación fundamental para consolidar funcionalidades, correcciones o avances realizados en ramas separadas.

---

## 2. Propósito

El merge se utiliza para:

- Integrar cambios de una rama a otra.
- Unificar trabajo desarrollado en paralelo.
- Mantener un historial consistente del proyecto.
- Incorporar nuevas funcionalidades a la rama principal.

Generalmente, el merge se realiza desde una rama principal como `main` hacia una rama secundaria o viceversa, dependiendo del flujo de trabajo.

---

## 3. Funcionamiento interno

Cuando Git ejecuta un merge:

1. Busca el **ancestro común** entre las ramas involucradas.
2. Analiza los commits realizados en cada rama desde ese punto.
3. Intenta aplicar automáticamente los cambios.
4. Crea un **commit de merge** solo si es necesario.

El comportamiento final depende de la relación entre los historiales de las ramas.

---

## 4. Tipos de merge básico

Existen dos situaciones comunes al realizar un merge:

### Merge Fast-Forward

- Ocurre cuando la rama destino no tiene commits propios.
- Git simplemente mueve el puntero de la rama.
- No se crea un commit adicional.

### Merge No Fast-Forward

- Ocurre cuando ambas ramas tienen commits diferentes.
- Git crea un commit de merge.
- Se conserva explícitamente el contexto del trabajo en ramas.

Estos casos se desarrollan con ejemplos prácticos en la carpeta `code/`.

---

## 5. Conceptos relacionados

- Rama (`branch`)
- Commit
- HEAD
- Historial de commits
- Fast-forward
- Conflictos de merge

---

## 6. Aclaraciones importantes

- El merge siempre se ejecuta **desde la rama destino**.
- Git decide automáticamente el tipo de merge.
- Las ramas no se eliminan después de un merge.
- Un merge puede requerir resolución manual de conflictos.

---

## 7. Ejemplo conceptual

Supongamos:

- Rama `main`: versión estable del proyecto.
- Rama `feature-login`: desarrollo de una nueva funcionalidad.

Flujo típico:

1. Se trabaja en `feature-login`.
2. Se vuelve a la rama `main`.
3. Se realiza el merge.
4. Los cambios quedan integrados en `main`.

---

## 8. Buenas prácticas

- Verificar la rama actual antes de mergear.
- Mantener el working tree limpio.
- Entender el historial antes de ejecutar el merge.
- Probar el proyecto luego del merge.
- Resolver conflictos revisando cuidadosamente el código.

---

## 9. Errores comunes

- Mergear desde la rama incorrecta.
- No comprender por qué Git crea o no un commit.
- Forzar merges sin entender el historial.
- Ignorar conflictos de merge.

---

## 10. Conceptos clave

- El merge une ramas.
- El historial determina el tipo de merge.
- Git automatiza la integración de cambios.
- Los conflictos son parte normal del flujo de trabajo.

---

## 11. Resumen final

El **merge básico** es una herramienta esencial en Git que permite integrar el trabajo realizado en distintas ramas.  
Comprender cómo funciona y qué tipo de merge aplica Git es clave para mantener un proyecto ordenado y estable.
